§2

Almide

“The language LLMs can write most accurately.”
LLM のために作った言語が、WASM の申し子だった。

Almide — LLM コード生成に最適化した言語

  • 静的型付け・関数型。Rust と WASM のデュアルターゲット
  • stdlib 834 関数 / 39 モジュール、テスト 2,681、ICE 0
  • native と wasm の観測可能挙動は契約で同一保証
  • 曖昧さを削る: 1 概念 1 記法、|> パイプライン、fan 並行
effect fn main() -> Unit = {
  let (sum, product) = fan {
    slow_add(3, 4)   // 並行実行
    slow_mul(3, 4)   // Err なら fail-fast
  }
  println("sum = " + int.to_string(sum))
}

// パイプラインが第一級のイディオム
fn parse_row(line: String) -> List[String] =
  string.split(line, ",") |> list.map((c) => string.trim(c))

設計の物差しは MSR — AI の修正に生き残る率

診断フィードバック付き最大 3 リトライ後に、コンパイルとテストが通った率。30 タスクを日次 CI で計測。

出典: almide-dojo（2026-07-02 のデイリー run、Claude は 29/30）。8B の小型モデルでも 4 割超が診断ループで生還する。

GC なし・注釈なし・リークなし — を機械証明

theorem perceus_all_heap_freed (fb : FnBody) :
    allHeapFreed (perceusTransform fb)
  • メモリ管理は Perceus 参照カウント。GC ポーズなし、Rust のような所有権注釈もなし
  • 「どんなプログラムでも、全ヒープ確保が全実行経路で解放される」ことを Lean 4 が検証（22+ theorems, sorry 0）
  • 紙の上の証明ではない — コンパイラの verify パイプラインで実行され、CI が未証明のマージをブロックする

効果システム × WASM — サンドボックスが二層になる

ビルド時（型）
effect 7 カテゴリを自動推論
IO / Net / Env / Time / Rand / Fan / Log を stdlib 使用から推論。
almide.toml の [permissions] にない capability はビルドエラーになる。
実行時（WASM）
capability サンドボックス
WASI import をロード時に検証し、宣言外の API はそもそもリンクされない。ランタイムが 2 枚目の壁になる。
「left-pad が fs を読めない世界」を、型とランタイムの両方で。

まとめ — WASM は次の Docker

  • WASM は「ブラウザの技術」から「言語非依存のユニバーサル実行単位」になった — 8 クラスタで本番稼働中
  • おもちゃではない — Almide 製の実用アプリ 6 本 + ライブラリ 3 本（約 13,600 行）が今日も動いている
  • LLM 時代の安全実行基盤は WASM が本命 — tiny binary × effect × capability はそのための組み合わせ
